# HMOG 五动作原始触摸轨迹：数据结构与预处理说明

## 1. 目标与边界

本目录新增的是“触摸轨迹数据层”，不是 IMU 生成器，也不是把 IMU mask 当成
触摸轨迹。

它完成两件事：

1. 直接复用 `1_data_processing/preprocess.py` 已审计的五类事件构造和互斥顺序；
2. 将这些派生事件重新对齐到 HMOG 原始 `TouchEvent.csv`，恢复真实的
   DOWN / MOVE / UP 时间轴、坐标、压力、接触面积和 pointer 信息。

五类 action id 不变：

| id | action |
| ---: | --- |
| 0 | tap |
| 1 | scroll |
| 2 | swipe（HMOG StrokeEvent / fling-style 项目标签） |
| 3 | pinch |
| 4 | keystroke |

轨迹层不使用 IMU 固定窗口的百分位裁剪、padding 或每 session 200 个窗口预算。
这些规则服务于固定长度 IMU 训练，不应删除或截断原始触摸 contact。

## 2. 原始 TouchEvent 字段

HMOG `TouchEvent.csv` 的列为：

```text
SystemTime, EventTime, ActivityID,
PointerCount, PointerID, ActionID,
X, Y, Pressure, ContactSize, Orientation
```

其中：

- `EventTime` 是本处理使用的权威相对时间，单位为毫秒；
- `ActionID=0/1/2/5/6` 分别对应 DOWN / UP / MOVE /
  POINTER_DOWN / POINTER_UP；
- 同一 `ActivityID + EventTime` 下的多行属于同一个 Android MotionEvent，
  每个活动 pointer 一行；
- 完整物理 contact 从主 pointer 的 DOWN 开始，到主 pointer 的 UP 结束；
- CANCEL、缺 DOWN、缺 UP 的 contact 不进入训练数据，只进入 audit。

原始触摸的实测典型帧间隔约为 16–17ms。脚本不把触摸重采样成 100Hz。
100Hz 是当前 IMU 输出频率，不能反过来改写原始触摸采样率。

## 3. 五类轨迹的严格语义

### 3.1 tap / scroll / swipe

保留事件必须对齐到一个完整且纯单指的 DOWN–UP contact：

- `max_pointer_count == 1`；
- `n_pointers == 1`；
- 原始 contact 内任何时刻出现第二根手指，整条拒绝；
- 不会为了增加数量而用起终点插值补出中间轨迹。

`scroll` 的派生 `CurrentTime` 往往停在最后一次 scroll callback，原始手指稍后才
UP。因此同时保存：

- `label_start_ms / label_end_ms`：派生 action 的有效区间；
- `touch_start_ms / touch_end_ms`：完整物理 DOWN–UP 区间。

### 3.2 pinch

派生标签仍严格来自 `(ActivityID, PinchID)` 的 phase 0 到 phase 2。

但输出轨迹保留包含这个 phase 的完整多指 contact，包括可能存在的单指 lead-in
和 lead-out。保留条件为：

- 完整 contact 至少两个 pointer；
- phase 0–2 active 区间内确实同时存在两个 pointer；
- 不用 200ms 间隔猜测 pinch 分组。

这样既保留 Android 合法的完整手势，又能用 `flat_active_mask` 精确指出真正的
pinch phase。

### 3.3 keystroke

先按已修正的 HMOG 定义构造按键：

```text
PressType=0 -> DOWN
PressType=1 -> UP
权威时间轴 -> PressTime / EventTime
```

完整 typing event 的每个 key 再按 `(ActivityID, DOWN, UP, orientation)` 对齐到
一个原始单指 contact。

关键语义：

- 一次 typing event 是多个离散 key contact 的有序序列；
- 每个 key 保留自己的 DOWN / MOVE / UP 坐标；
- `flight = next_DOWN - previous_UP`；
- 手指离开屏幕的 flight 期间没有 XY，因此不会伪造“从上一个字母滑到下一个
  字母”的屏幕直线；
- strict 默认下，只要 typing event 中一个 key 缺原始 contact，就拒绝整个
  typing event，避免部分事件改变字母数、hold 和 flight 分布；
- `--allow-partial-keystroke` 只用于诊断，不应作为正式训练默认值。

## 4. 为什么还会拒绝已经通过五类互斥的事件

派生 callback 区间互斥不等于物理 DOWN–UP contact 一一对应。

例如，用户先用一根手指移动，随后第二根手指落下开始 pinch：

```text
完整 raw contact:  DOWN ---- one-finger move ---- two-finger pinch ---- UP
派生 scroll:       [--------------------)
派生 pinch:                              [---------------------)
```

两个半开区间可以在 phase 边界不重叠，但它们共享同一个物理 contact。若同时把
完整 contact 写入 scroll 和 pinch，会造成跨类别重复与数据泄漏。严格轨迹层按
优先级 `keystroke > pinch > swipe > scroll > tap` 只保留一次，并在 audit 中记为：

```text
raw_contact_reserved_by_higher_priority_event
```

这不是时间匹配失败，也不是为了减少样本而抽稀。

## 5. NPZ：numeric-only flat + offset

每类一个文件：

```text
hmog_trajectory_tap.npz
hmog_trajectory_scroll.npz
hmog_trajectory_swipe.npz
hmog_trajectory_pinch.npz
hmog_trajectory_keystroke.npz
```

所有数组均为固定数值 dtype 或固定 Unicode scalar，不含 object array，不使用
pickle。读取方式：

```python
with np.load(path, allow_pickle=False) as data:
    i = 10
    left = data["event_offsets"][i]
    right = data["event_offsets"][i + 1]
    x = data["flat_x"][left:right]
    y = data["flat_y"][left:right]
    t = data["flat_event_time_ms"][left:right]
```

### 5.1 event offsets

| 数组 | shape | 含义 |
| --- | --- | --- |
| `event_offsets` | `[E+1]` | 第 `i` 个事件在所有 `flat_*` 数组中的 `[left,right)` |
| `event_key_offsets` | `[E+1]` | 第 `i` 个 typing event 在所有 `key_*` 数组中的范围 |
| `key_touch_offsets` | `[K+1]` | 第 `k` 个 key 在 `flat_*` 中的触摸点范围 |

三组 offset 都必须：首项为 0、单调不减、末项精确等于对应 flat/key 总长度。
每个保留 event 和 key 都至少有一行，因此正式输出中的对应区间为非空。

### 5.2 flat touch arrays

所有 `flat_*` 数组长度相同，等于 flat rows 总数：

| 字段 | dtype | 含义 |
| --- | --- | --- |
| `flat_system_time_ms` | int64 | 原始 SystemTime |
| `flat_event_time_ms` | int64 | 原始 EventTime，毫秒 |
| `flat_t_rel_ms` | int64 | 相对本完整 contact / typing event 起点时间 |
| `flat_frame_index` | int32 | event 内 MotionEvent 帧编号；pinch 同一帧有两行 |
| `flat_pointer_count` | int8 | 当前帧 pointer 数 |
| `flat_pointer_id` | int16 | pointer id |
| `flat_action_code` | int8 | Android actionMasked |
| `flat_x`, `flat_y` | float32 | 未缩放的原始屏幕坐标 |
| `flat_pressure`, `flat_size` | float32 | 原始压力和接触面积 |
| `flat_orientation_id` | int8 | 该触摸行的 orientation |
| `flat_active_mask` | uint8 | 是否处于派生 action active 区间 |
| `flat_valid_mask` | uint8 | flat row 是否有效；当前严格输出全部为 1 |
| `flat_key_index` | int32 | typing event 内 key 下标；非 keystroke 为 -1 |
| `flat_keycode` | int32 | 当前 keycode；非 keystroke 为 -1 |

offset 已定义有效长度，`flat_valid_mask` 是显式一致性字段，不用于 padding。
`flat_active_mask` 与 IMU 的动作 mask 语义相似，但它位于原始不规则触摸点上，
不能直接当作 100Hz IMU mask。

### 5.3 event arrays

event 数组长度均为 `E`，核心字段包括：

| 字段 | 含义 |
| --- | --- |
| `event_id` | 由全局 user index、session、action、action 内序号生成的稳定 id |
| `user_id` | HMOG 外层 zip 排序后的全局 index；抽一个用户时也不重新编号 |
| `user_external_id` | HMOG 原始匿名数字用户 id |
| `session_id`, `activity_id`, `action_id` | 原始来源与标签 |
| `orientation_id` | 已审计事件的生成条件方向 |
| `label_start/end/duration_ms` | 派生 action active 区间 |
| `touch_start/end/duration_ms` | 完整 raw contact 区间 |
| `active_start/end_rel_ms` | 原始 label active 区间相对完整 contact 起点的位置；因 25ms 匹配容差可轻微小于 0 或大于 touch duration |
| `n_rows`, `n_frames`, `n_pointers`, `max_pointer_count` | 轨迹结构统计；keystroke 的 pointer 数是“每个独立 key contact 同时为单指”，不是跨整段统计历史 tracking ID 的 unique 数 |
| `raw_gesture_id`, `n_raw_gestures` | 原始 contact 溯源；keystroke 为多个 contact |
| `match_start/end_error_ms` | 对齐误差；keystroke 为 key 误差的均值 |
| `n_keys`, `n_letters` | typing event 条件；非 keystroke 为 0 |
| `motion_id_raw`, `posture_id`, `task_id`, `activity_subtask_id` | Activity metadata |
| `meta_*` | 已审计 event 中保存的 XY、span、speed、pressure 等 metadata |

`active_start/end_rel_ms` 不会为了落进 raw contact 而裁剪，因为裁剪会悄悄改写原始
label 时轴。正式审计强制它们与 `match_start/end_error_ms` 精确对应，并要求 label 与
touch 区间有非空交集。逐点是否真的落在 active 区间由 `flat_active_mask` 表示，因此
区间外不存在被虚构出来的触摸点。

Keystroke 是多个独立 DOWN..UP contact 的 contact-major 存储。后一个 key 可获得新的
Android pointer tracking ID；这不代表同一时刻有两指。每个 key 单独强制恰好一个
pointer、完整 DOWN/UP。HMOG 还存在合法的 zero-flight：上一 key UP 与下一 key DOWN
可共享同一毫秒，因此跨 key 的全局时间是 non-decreasing；每个 key contact 内仍严格
递增，禁止把同毫秒边界误删或伪造一个 midpoint gap。

### 5.4 key arrays

`key_*` 只在 keystroke 文件中非空：

- `key_index_in_event`, `keycode`, `key_is_letter`；
- `key_down_ms`, `key_up_ms`, `key_hold_ms`；
- `key_flight_from_previous_ms`；
- `key_orientation_id`, `key_raw_gesture_id`；
- `key_touch_start/end_ms`；
- `key_match_start/end_error_ms`, `key_touch_found`。

## 6. manifest 与 audit

| 文件 | 用途 |
| --- | --- |
| `manifest.json` | 输入路径、已审计 preprocessor SHA256、配置 SHA256、用户选择、匹配参数、每个 NPZ 数量/大小/SHA256 |
| `audit.json` | action 接收/拒绝计数、拒绝原因、raw parser 总计与逐 session 统计 |
| `event_audit.csv` | 每个候选 event 的状态、原因、时间误差、pointer 数和 key 对齐数 |

正式训练必须保留 audit；不能只看最终数量后删除拒绝原因。

## 9. 已知限制

1. HMOG 的派生 callback 并不保证每条都能在 raw TouchEvent 中找到完整 contact；
   strict 版本选择拒绝，绝不补造。
2. 一次完整 contact 可能先触发 scroll，后触发 pinch；轨迹层为避免跨类重复只保留
   高优先级完整 contact。
3. 屏幕像素坐标尚未归一化；训练时应结合设备尺寸与 orientation 做确定性映射，
   同时保留 raw 坐标用于审计。
4. 该预处理只恢复真实轨迹，不代表已经完成轨迹生成模型、Android Type-B 多指注入
   或 AHB 检测器训练。
