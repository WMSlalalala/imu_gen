## 3. 轨迹目标协议

统一流程为：

```text
shared EventPlan
  ├─ IMU generator        ──> IMU detector
  └─ trajectory generator ──> trajectory detector
                  两路同一事件 ──> cross-modal consistency ──> total detector
```

### 3.1 完整轨迹

- 动作：`tap / scroll / swipe / pinch / keystroke`；
- 时间：不规则毫秒时间戳，不把轨迹强制成固定长度；
- 单指：保留 down/move/up 和 pointer lifetime；
- pinch：双指共用全局 union clock，不能拆成两条各自归一化的轨迹；
- keystroke：保留离散 contact、hold、flight、keycode 和无接触 gap；
- 生成条件由 shared EventPlan 固定，IMU 与 trajectory 必须共享 action、duration、事件身份和随机种子关系。

### 3.2 Five-shot

- 每个 user/action 固定恰好 5 条真实参考；
- reference registry 只由 train 数据生成；
- 同一正式 run 内引用集合固定，不按生成结果重新挑选；
- five-shot 是条件先验，不允许直接重放模板，也不允许用 test 结果选择参考。
- 正式 condition preflight 已穷举 100,000 个请求并证明固定五 refs、条件/噪声种子唯一且域不相交；正式生成后 `audit_five_shot_generation.py` 还会逐个重载 500 个 user/action 单元、重算 registry SHA、要求每单元恰好 200 条且五动作合计 100,000 条，并把 500 个 NPZ 的字节哈希闭包到最终 receipt；
- trajectory 与 IMU 使用各自模态的固定五参考：共享的是 EventPlan 的用户、动作、时长、方向、事件身份和种子关系，不伪称两路 reference ID 指向同一物理原始行。paired IMU 审计另外要求 `used_ref_indices` 为长度 5、互异且同一 user/action 固定。
- 2026-07-22 07:11 EDT 再读正式证据：condition preflight 的五动作各20,000请求、100用户×200条，`fixed_references_per_user_action=5`，condition/noise seed与fake ID均全局唯一且两种seed域不相交，`train_prior_only_fixed_train_users=true`、no retry/no skip。已启动的tap/scroll/swipe实际`reference_audit.json`在train/val/test均passed，cross-user、cross-split、duplicate、target-in-ref均为0，test未用于训练或选择，其registry SHA逐项匹配preflight；pinch/keystroke的registry SHA虽已由preflight固化，但实际训练目录尚未物化，因为对应动作尚未启动，因此不能提前写成训练完成。

### 3.3 不同时长

- 生成侧：EventPlan 显式携带 duration/timeline，覆盖动作真实时长分布，不用一个固定长度代表全部事件；
- 检测侧：保留原始 duration，并报告总体与 duration-stratified 指标；
- 分箱边界只根据 train duration 的分位数拟合；
- detector threshold 仍然只从 validation 全局选择；每个时长箱只应用该固定阈值，严禁按 test 时长箱重新调阈值；
- 重复时长导致相邻分位点相同时，合并为空间可达的唯一 cut，报告 requested/effective bins。
- 正式 100,000-condition preflight 已实际覆盖非固定时长：tap `14–354 ms`、scroll `130–6255 ms`、swipe `59–6260 ms`、pinch `151–2892 ms`、keystroke `438–41590 ms`；keystroke 20,000 个请求全部走 hard-timeline projection，并逐 key/event 核对 contact/hold/flight。这里证明的是生成条件空间与共享 EventPlan 的时长覆盖，不是100,000条生成轨迹已经完成，也不替代后续25个基础轨迹检测器和5个综合检测器的train-only时长分箱及val/test分层指标。

### 3.4 指标

轨迹检测器和综合检测器均统一 fake-high：`score < threshold` 接受为 real。正式报告至少包括：

- validation 选出的 EER operating point；
- validation FRR <= 5% operating point；
- test FA、FRR、AUC 和 FA-FRR curve；
- 500 次 seeded user-level bootstrap CI；
- 每个 duration bin 的样本量、real/fake 数、AUC、固定阈值下的 FA/FRR；
- 运行时 latency；
- coverage、schema、样本身份、用户 split、reference registry 和 checkpoint hash 审计。
