# IMU + trajectory 总检测器与更像人的联合生成方案

## 1. 结论

下一步不能把 IMU 和 trajectory 各自独立生成后再随便拼起来。那样总检测器会很容易抓到跨模态不一致：

- trajectory 的 duration 和 IMU active_len 对不上；
- swipe/scroll 的位移速度和 IMU 能量不匹配；
- pinch 的双指 span/angle 变化和 IMU 旋转/加速度不匹配；
- keystroke 的字母数量、hold/flight 和 IMU 窗口长度不匹配。

正式版应先采样同一份 event metadata，再同时驱动 trajectory 和 IMU：

```text
user/action/text/xy/duration/orientation
        ↓
shared event plan
        ├── trajectory generator
        └── IMU generator
        ↓
paired sample_id
        ↓
trajectory detector + IMU detector + total detector
```

## 2. “模拟得像一点”的具体做法

### 2.1 先生成 event plan

每条样本先确定：

- `sample_id`
- `user_id`
- `action`
- `orientation_id`
- `duration_ms`
- `xy_start / xy_end`
- `distance / direction`
- pinch 的 `start_span / end_span / angle`
- keystroke 的 `text / n_keys / n_letters / hold / flight`

这些 metadata 只能来自 train users 的 empirical prior 与该用户 few-shot refs，不能从 test 统计里抽。

### 2.2 trajectory 按 plan 生成

轨迹生成不是简单直线加噪声。需要保留人类触摸的几个约束：

- 时间轴保留 HMOG/Android 的不规则 touch event 间隔；
- 单指动作使用人类 velocity profile，而不是匀速；
- tap 保留微小触点抖动；
- scroll/swipe 保留路径弯曲、局部停顿和末端减速；
- pinch 保留两指不同 DOWN/UP 生命周期、span/angle 变化；
- keystroke 保留每个 key 独立 DOWN/MOVE/UP，不插值成连续滑动；
- 每条 fake 检查 exact replay，不能直接复制 refs。

### 2.3 IMU 按同一 plan 生成

IMU 不再单独抽 duration/xy。它使用同一份 event plan：

- `duration_ms → active_len/mask`
- `orientation_id → 方向条件`
- `xy/distance → motion intensity condition`
- `n_letters/n_keys → keystroke condition`

这样 detector 同时看 trajectory 和 IMU 时，不会看到两个模态互相矛盾。

## 3. 总检测器定义

总检测器检测的是同一条样本的两路输出：

```text
IMU detector outputs + trajectory detector outputs
        + touch/IMU physical consistency features
        → total PAD score
```

已实现第一版：

- 代码：`estimator/total_detector.py`
- smoke：`scripts/smoke_total_detector.py`

输入是一行 paired feature，例如：

```text
imu detector score
imu score - validation threshold
imu rejected flag
trajectory detector score
trajectory score - validation threshold
trajectory rejected flag
touch speed 与 IMU acceleration/gyro correlation
touch peak 与 IMU motion peak 的时间差
path length / point rate / pointer count / n_keys
```

正式 total detector 默认强制要求 `consistency__*` 字段；只有旧 artifact/smoke 可以显式使用
score-only 兼容模式。因此它不再只是一个 detector score 拼接器。

然后训练一个二阶段 detector：

- 默认：`logistic`
- 可选：`linear_svm / rbf_svm`

它仍然遵守同一协议：

- train rows 只训练 total detector 和 scaler；
- validation rows 只选 EER / FRR<=5% 阈值；
- test rows 只汇报 FA / FRR / AUC；
- score 越大越像 fake；
- `score < threshold` 才接受为 real。

## 4. 为什么必须有 paired sample_id

总检测器不能用“第 1 条 IMU fake + 第 1 条 trajectory fake”这种方式硬拼。

正式要求：

```text
sample_id 相同
user_id 相同
action 相同
pool 相同
label 相同
```

只有这样，IMU 和 trajectory 才代表同一次事件。

如果旧结果没有共同 sample_id，只能做诊断版 ensemble，不能作为正式主结果。

## 5. 正式测试输出

正式总检测器要输出三组结果：

1. IMU-only；
2. trajectory-only；
3. IMU+trajectory total。

每组都按五动作报告：

- FA at validation EER threshold；
- FRR at validation EER threshold；
- AUC；
- FA at validation FRR target；
- FRR at validation FRR target；
- FA-FRR test curve；
- user-level bootstrap CI。
