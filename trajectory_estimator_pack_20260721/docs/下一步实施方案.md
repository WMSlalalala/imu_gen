# 下一步：参考 AHB benchmark 的 trajectory 估计/检测封装方案

## 1. 目标

这一步不是改 IMU 生成器，也不是把 detector critic 混到生成器里。

目标是把“轨迹是否像人”的估计能力封装出来，形成一个可调用模块：

```text
Android/Type-B 触摸轨迹
        ↓
轨迹合法性检查与特征提取
        ↓
benchmark-style detectors
        ↓
fake-high score + threshold decision + latency
```

输出和之前 IMU PAD 一样统一：

- `score`：越大越像 fake；
- `score < threshold`：接受为 real；
- threshold 只从 validation 选；
- test / runtime 只应用固定 threshold；
- 同时输出 EER operating point 和 validation FRR<=5% operating point。

## 2. 参考的 benchmark 检测方式

参考对象：

- GitHub：<https://github.com/Gebro13/Passing-the-Turing-Test-on-Screen-Agent-Humanization-Benchmark>
- Paper：<https://arxiv.org/abs/2604.09574>

上游 AHB 的核心检测思想是：从 Android touch 动作中抽取轨迹动力学特征，然后用 SVM / XGBoost 等分类器区分 human 和 agent。README 也明确写了其 analysis pipeline 包含特征提取、训练分类器、评估 humanization。

本项目不直接复用上游不严格的随机切分结果，而是保留它的检测思路，改成我们之前已经冻结的严格协议：

1. 五动作：`tap / scroll / swipe / pinch / keystroke`；
2. feature PAD：AHB-style 轨迹统计特征；
3. deep PAD：raw trajectory sequence TCN / Transformer；
4. fake score 方向统一为 fake-high；
5. scaler/model 只在 train 拟合；
6. threshold 只在 validation 选择；
7. test 只做固定阈值评估和 FA-FRR 曲线；
8. runtime estimator 只加载已导出的 detector artifact，不训练、不调阈值。

## 3. 为什么要单独封装“估计器”

之前 benchmark 代码主要是离线完整评估：

```text
dataset bundle → train detectors → validation 选阈值 → test 汇报 FA/FRR/AUC
```

但 Android 封装层还需要一个运行时接口：

```text
给一条将要注入/已经生成的 trajectory
        ↓
快速估计它在 benchmark detector 下像不像 fake
```

所以新增 estimator 封装层。它和正式检测器共享同一套模型、特征和阈值，但调用粒度从“整批实验”变成“单条或小批量轨迹”。

## 4. 输入格式

运行时标准输入是 `RawTrajectoryRecord`，也支持一个简单 NPZ：

```text
action
label              # runtime 可填 0，占位用；不参与 score
user_id
pool               # runtime 可填 test，占位用；不参与 score
sample_id
pointer_continuous # [2,T,4]，x/y/pressure/size
global_t_ms        # [T]，事件全局时间轴，不按 pointer 单独归一化
contact_mask       # [2,T]
active_mask        # optional
action_code        # optional，Android actionMasked；缺省由 contact 填 neutral
keycode            # optional，keystroke 用 canonical HMOG keycode
event_ids          # optional，keystroke 每个 key 一个 event_id
gap_mask           # optional，keystroke positive flight no-contact token
```

pinch 必须保留双指共享全局时间轴；keystroke 必须保留离散 key contact，不能插值成一条连续滑动。

## 5. Feature estimator

Feature estimator 已封装：

- 代码：`estimator/feature_estimator.py`
- 导出：`scripts/train_feature_estimator_artifacts.py`
- smoke：`scripts/smoke_estimator_pack.py`

它做的事：

1. 从 `RawTrajectoryRecord` 提取冻结特征；
2. 加载 train-only `StandardScaler`；
3. 加载 sklearn/xgboost detector；
4. 输出 fake-high score；
5. 用 validation 阈值给出 `accepted_as_real / rejected_as_fake`。

特征定义不重新写，直接调用旧 benchmark 工程里的冻结函数：

- `tap / scroll / swipe`：单指 AHB-style 24 维；
- `pinch`：center path 24 维 + span/angle/two-finger dynamics；
- `keystroke`：key count、letter count、hold、flight、burst、key transition distance。

注意：旧 feature PAD 正式输出只有 `summary.json / score_dump.npz / curves.npz`，没有保存 sklearn/xgboost 模型。因此这一步新增 `artifact.joblib`，否则无法运行时封装估计。

## 6. Deep estimator

Deep estimator 已写加载接口：

- 代码：`estimator/deep_estimator.py`

它直接加载之前 raw-sequence Deep PAD 的：

- `summary.json`
- `checkpoints/best_epoch_*.pt`
- checkpoint 内保存的 train-only normalizer
- checkpoint 内保存的模型结构参数

推理时：

```text
RawTrajectoryRecord
        ↓
collate_raw_sequences + checkpoint normalizer
        ↓
TCN / Transformer
        ↓
sigmoid(logit) = fake-high score
        ↓
validation threshold decision
```

Deep estimator 不复用 generator critic，也不读取 test 阈值。

## 7. Unified service

统一入口：

```python
from estimator.service import TrajectoryEstimatorService

service = TrajectoryEstimatorService.load("estimator_manifest.json")
result = service.estimate_record(record)
```

输出结构：

```json
{
  "schema_version": "trajectory_estimator_result_v1",
  "action": "swipe",
  "score_direction": "fake_high",
  "acceptance_rule": "score < threshold",
  "detectors": [
    {
      "family": "feature_pad",
      "detector": "rbf_svm",
      "score": 0.123,
      "decisions": {
        "eer": {"accepted_as_real": true},
        "val_frr_le_5pct": {"accepted_as_real": false}
      },
      "latency_ms": 0.4
    }
  ],
  "ensemble": {
    "eer": {
      "fake_vote_rate": 0.25,
      "accepted_by_majority": true
    }
  },
  "latency_ms": 1.2
}
```

这里的 ensemble 只做投票，不把 SVM decision score 和 XGBoost probability 直接平均，因为不同 detector 的 raw score 标尺不同。

## 8. 后续正式流程

### Phase A：artifact export

输入之前完整 trajectory detector bundle：

```bash
python scripts/train_feature_estimator_artifacts.py \
  --dataset-dir <trajectory_pad_bundle_dir> \
  --fake-user-split /home/mwang49/real-human/imu_gen/final/data/splits/users_seed42.json \
  --output-dir <run_dir>/feature_artifacts \
  --bootstrap-replicates 500
```

产物：

- `estimator_manifest.json`
- `feature_pad/<action>/<detector>/artifact.joblib`
- `summary.json`
- `score_dump.npz`
- `curves.npz`

### Phase B：deep detector artifact binding

如果已有 benchmark deep 结果，则把每个：

```text
deep_pad/<action>/<detector>/summary.json
deep_pad/<action>/<detector>/checkpoints/best_epoch_*.pt
```

写入同一个 `estimator_manifest.json` 的 `deep_pad` 字段。

### Phase C：runtime query

Android 轨迹生成/注入接口输出 Type-B 行后，转换为 `RawTrajectoryRecord`，再调用：

```bash
python scripts/estimate_trajectory.py \
  --manifest <estimator_manifest.json> \
  --record-npz <one_record.npz> \
  --feature-only
```

如果 deep checkpoint 可用，去掉 `--feature-only` 即可同时跑 deep。

### Phase D：formal benchmark

正式报告仍然按之前 benchmark 协议跑全量：

- fake train/val/test users = 70/10/20；
- real train/val/test = 每个 user/action 内 event-group 60/20/20；
- threshold from validation；
- test 汇报 FA、FRR、AUC、FA-FRR 曲线；
- user-level bootstrap 只做 CI，不改主结果。
